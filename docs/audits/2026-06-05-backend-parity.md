# Backend Parity: apps/server vs template-express

`apps/server` is a substantial extension of `template-express` and correctly implements most of the same foundational decisions: layered architecture, factory-based DI lifted to a flat-export pattern, Pino structured logging, Zod schemas as source of truth, SHA-256 session token hashing, bcrypt password storage, explicit-column repository queries, graceful shutdown, health endpoints, CORS with credentials, Helmet, request ID propagation, and the two-file entry point split. The biggest category of drift falls into three buckets: (1) three ADRs from the reference are entirely absent from the target -- idempotency keys, a `validate` middleware extracted to its own module, and a `helpers/` layer; (2) the error handler lacks the 503-vs-500 database-error distinction; and (3) the routes are not versioned under `/v1/`. The `posts` example resource and its migration, handler, schema, and tests are fully absent and need to be ported in. Reconciliation is achievable incrementally; the complex items are the `posts` port and the error-handler database-detection addition.

---

## Owner carve-outs (intentional, not divergences)

1. **Per-folder file layout.** `handlers/auth/auth.ts`, `middleware/csrfGuard/csrfGuard.ts`, etc. The reference's flat files (`handlers/auth.ts`) are not the target model. New files added for the `posts` port must follow the per-folder convention.
2. **node-pg-migrate migrations.** The reference uses raw `.sql` files and a custom `migrate.ts` runner. The target's `node-pg-migrate` + `.js` builder API is preferred. Any new migration (including `posts`) must be authored as a `node-pg-migrate` migration.
3. **Integration test location `src/__tests__/integration/`.** The reference uses `src/__integration__/`. The target location is preferred.
4. **`posts` resource: add, not remove.** The absence of `posts` is a divergence to fix by adding the resource (per-folder layout, node-pg-migrate migration, tests under `__tests__/`). Nothing is removed.
5. **Extra surface preserved.** Stripe/billing, Cloudflare R2, Redis, BullMQ worker, Sentry, PostHog, CSRF guard, circuit-breaker, and the Next.js frontend are intentional additions. Do not flag for removal.

## Structural and convention divergences

### S-01: Missing `helpers/` layer

**Reference:** `src/helpers/auth.ts` (creates session cookie), `src/helpers/hash.ts` (`hashToken`).
**Target:** No `src/helpers/` directory. `hashSessionToken` is a private function inside `repositories/auth/auth.ts`. Session cookie creation logic lives in `handlers/auth/auth.ts` as `sessionCookieOptions()` and `clearCookieOptions()`.
**Fix:** Create `src/utils/crypto/hash.ts` (or `src/helpers/hash.ts`) exporting `hashToken`. Move `hashSessionToken` there. The session cookie helpers can remain in the handler since the target's CLAUDE.md documents that pattern.

### S-02: Missing `validate` middleware

**Reference:** `src/middleware/validate.ts` -- a reusable `validate(schema, source)` middleware that runs `schema.safeParse(req[source])`, replaces the request field with coerced data, and returns 400 with the first error message. Used in every route definition.
**Target:** Each handler calls `schema.safeParse(req.body)` inline. This works but duplicates the parse-fail response logic (eight or more `if (!parsed.success) { res.status(400)... }` blocks across handlers).
**Fix:** Add `src/middleware/validate/validate.ts` matching the reference implementation, wire it in routes. This reduces handler boilerplate and centralizes the 400 response shape.

### S-03: No `errors.ts` error-code registry

**Reference:** `src/errors.ts` exports `ERROR_CODES` (nested by domain), `ErrorCode` type, `ErrorResponse` interface, and `createErrorResponse` factory. Every error response has both `code` and `error` fields.
**Target:** No central error registry. Error responses are `{ error: { message: '...' } }` inline strings. Clients cannot switch on a stable code. If the target is to remain client-friendly, add `src/errors.ts` and enforce the `{ code, error }` shape everywhere. (The response shape change is breaking if clients exist.)
**Files to create:** `apps/server/src/errors.ts`.
**Files to update:** All handler files, `notFoundHandler`, `csrfGuard`, `requireAuth`, `rateLimiter`.

### S-04: Rate limiter not bypassed in test environment

**Reference:** `app.ts` wraps rate limiters in `if (!config.isTest)` blocks.
**Target:** Rate limiter always active. Tests that hit routes will be throttled after enough requests.
**Fix:** Add `isTest` export to `src/config/env.ts` (`export const isTest = env.NODE_ENV === 'test'`), then wrap `app.use(rateLimiter)` in `if (!isTest)` in `app.ts`. Auth routes: apply the same guard on `authRateLimiter` application.

### S-05: `SALT_ROUNDS` defined in two places

**Target:** `SALT_ROUNDS = 12` is defined independently in both `repositories/auth/auth.ts` (for `createUser`) and `handlers/auth/auth.ts` (for `resetPassword` and `updateMe`).
**Fix:** Extract to `src/constants/auth.ts` (or `src/config/auth.ts`) and import from both. One value to update if rounds ever change.

### S-06: `SELECT *` in billing repository

**Reference:** Post columns use a `POST_COLUMNS` constant; no `SELECT *`.
**Target:** `repositories/billing.ts` uses `SELECT *` for subscription lookups.
**Fix:** Define a typed column list constant and replace the `SELECT *` in `getSubscriptionByUserId` and `getSubscriptionByStripeSubscriptionId`.

### S-07: Logger not silenced in test

**Reference:** Logger uses `level: config.isTest ? 'silent' : 'info'`.
**Target:** Logger sets level to `debug` in non-production, `info` in production. No silence in test. Test output may contain pino log lines.
**Fix:** Add `level: isTest ? 'silent' : isProd ? 'info' : 'debug'` in `src/utils/logs/logger.ts`.

### S-08: pg_cron session cleanup migration absent

**Reference:** `20260603-1400-schedule-session-cleanup.sql` registers an hourly pg_cron job that deletes sessions older than 7 days. The migration silently skips on databases without pg_cron.
**Target:** No equivalent migration. The `setInterval` cleanup in `app.ts` handles this at runtime but does not persist the schedule in the database.
**Fix:** Add a node-pg-migrate migration that attempts to register the pg_cron job with the same silent-skip pattern.

### S-09: Missing OpenAPI spec

**Reference:** `docs/openapi.yaml` covers all endpoints.
**Target:** No spec file.
**Fix:** Author `docs/openapi.yaml` for the target's routes. This is a documentation task, not a code change.

### S-10: `parsePagination` utility exists but is not wired

**Target has** `src/utils/parsers/parsePagination.ts` and tests. It is not used anywhere because the `posts` resource is absent.
Once the `posts` resource is ported, wire `parsePagination` in the posts router/handler to complete the pagination story.

### Wire-up changes

- `apps/server/src/app.ts`: add `import { postsRouter } from 'app/routes/posts.js'` and `app.use('/v1/posts', postsRouter)` (or `/posts` if versioning is deferred).
- Update `apps/server/src/__tests__/routes/routes.test.ts` to add posts route smoke tests.

### Data model adaptation note

The reference `posts` table references `users.email`. The target's `users` table uses uuid primary keys. The ported migration must use `user_id uuid NOT NULL REFERENCES users(id) ON DELETE CASCADE` and all repository queries must scope by `user_id`, not `email`.
